🔎 Surprises & Découvertes

⚡

GLM OCR : ultra-rapide, ultra-dangereux

En seulement 2,8 s et pour 0,005 ¢, GLM OCR produit un charabia technique : « débruit d’une trame » au lieu de « début d’une trame » et « Bos olds » au lieu de « Bas alors ». Ces deux erreurs changent complètement le sens d’une spécification I²C. Le gain de temps et de coût ne compense pas une transcription qui invente des mots.

🎯

Le trio gratuit qui copie sans faute

Gemma 4 31B, Qwen3 VL 30B et Qwen2.5 VL 72B produisent tous une transcription française正確e avec accents et termes techniques (« début », « caractérisée », « SDA », « SCL »), pour un coût total de 0 $. Qwen2.5 VL 72B atteint même 13,1 s pour 2018 tokens sans une seule coquille. Un rapport qualité/prix imbattable pour de l’écriture manuscrite technique.

📝

Qwen3.5 122B : le bavard qui lit les métadonnées

Au lieu de transcrire le texte manuscrit, ce modèle produit 1644 tokens décrivant l’image elle-même (« I will go through the document section by section »). Il n’extrait aucun contenu utile pour le protocole I²C. C’est le seul modèle du benchmark qui ignore la tâche demandée, ce qui le rend inexploitable en production malgré son temps correct de 5,6 s.

💻

Code C : la lenteur paie, la rapidité aveugle

Sur l’image WhatsApp contenant du code C, les modèles les plus rapides (Gemini 3.5 Flash : 2,1 s) échouent complètement (« no handwritten text »), tandis que Claude Sonnet 5 (6,9 s) et Qwen3 VL 235B (5,7 s) restituent parfaitement la fonction Regulation_FiO2 avec ses pointillés. La vitesse sans capacité de lecture contextuelle est un défaut rédhibitoire pour du code manuscrit.

💡 Recommandations

Précision Maximale

Qwen2.5 VL 72B + Gemma 4 31B

Ces deux modèles open‑source gratuits offrent la meilleure fidélité pour la transcription technique française. Qwen2.5 VL 72B restitue les accents et la casse (START, STOP) sans aucune erreur ; Gemma 4 31B excelle sur les formulaires structurés (BTS). Aucun coût d’API, une fiabilité proche de 100 % sur nos échantillons.

▶ À privilégier pour les examens, brevets et documents académiques.
Meilleur Rapport Qualité/Prix

Qwen3 VL 30B + GLM 4.6V

Qwen3 VL 30B (8,1 s, gratuit) transcrit parfaitement le protocole I²C. GLM 4.6V (35,4 s, gratuit) gère bien les tableaux HTML du document BTS. Le duo couvre à la fois le manuscrit courant et les mises en page complexes pour un coût nul. Léger défaut de GLM 4.6V sur la vitesse, mais la qualité est au rendez‑vous.

▶ Idéal pour les startups et les petits budgets.
Pipeline de Production

Claude Sonnet 5 + Qwen3 VL 235B

Claude Sonnet 5 (12 s, 1 ¢) est le seul à encapsuler le code C dans un bloc markdown et à respecter la ponctuation. Qwen3 VL 235B (8,8 s, 0,08 ¢) produit des transcriptions françaises fluides avec une seule anomalie mineure (« inactive » au lieu de « caractérisée »). Ensemble, ils couvrent le texte courant et le code avec une latence acceptable.

▶ À utiliser en relais : Qwen pour le brut, Claude pour la mise en forme.
Choix Équilibré

Qwen3.5 122B — avec précaution

Ce modèle est gratuit et rapide (5,6 s), mais il a tendance à décrire l’image plutôt qu’à transcrire le texte. Sur le document BTS, il réussit bien (extraction du titre et des sections). En revanche, sur l’I²C, il produit un long préambule inutile. Utilisable seulement si l’on valide en aval que le contenu est du texte brut et non une description.

▶ Ajouter un filtre « commence par du texte » dans le pipeline.

🔬 Modèles Non Testés

Ces modèles du benchmark AIMultiple nécessitent des APIs ou infrastructures dédiées

🧠

olmOCR-2-7B-1025-FP8

top 4

Non testé car nécessite un déploiement local avec GPU (modèle 7B en FP8). Pour l’essayer, lancez-le via Ollama ou une instance Hugging Face avec transformers et un T4+. Prometteur pour l’OCR léger.

🌙

Moondream OCR

milieu

Non testé – API propriétaire nécessitant une clé. Pour l’évaluer, inscrivez-vous sur le portail Moondream et utilisez leur endpoint /ocr avec vos images. Adapté aux transcriptions rapides en anglais.

🔍

DeepSeek OCR

bas

Non testé – accès via API DeepSeek (payant). Pour tester, créez un compte DeepSeek et appelez POST /ocr avec une image. Réputé bon pour le chinois et l’anglais, reste à vérifier pour le français manuscrit.

📦

PaddleOCR-VL

bas

Non testé – modèle open‑source à déployer via PaddlePaddle. Pour l’utiliser, suivez le guide d’installation et exécutez paddleocr --image_dir=. Excellent pour les documents multi‑langues.

☁️

Azure Cognitive Service

—

Non testé – API Azure nécessitant une clé et un compte. Pour tester, créez une ressource « Computer Vision » dans le portail Azure et utilisez l’endpoint /vision/v3.2/ocr. Robuste pour l’impression, inconnu pour le manuscrit.

🔮

Google Vision API

—

Non testé – API cloud avec authentification OAuth. Activez « Vision API » dans Google Cloud Console, puis appelez annotateImage(). Performant pour le texte imprimé, à évaluer sur écriture cursive française.

📄

Amazon Textract

—

Non testé – service AWS nécessitant des clés IAM. Pour l’essayer, utilisez la console AWS ou le SDK Python avec detect_document_text(). Optimisé pour les formulaires et les tableaux, à tester sur le document BTS.

🌀

Mistral OCR

—

Non testé – accès via API Mistral (clé requise). Inscrivez-vous sur console.mistral.ai, générez une clé et utilisez l’endpoint /v1/ocr. Modèle prometteur pour le français, à valider sur des échantillons manuscrits.
